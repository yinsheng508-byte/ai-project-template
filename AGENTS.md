# AGENTS.md

## 目标
保持 AI 协作开发过程清晰、可控、可 review、可回滚。

## 核心协作原则
1. 先把项目当前正式文件全部提交到 `main`，再开始并行开发
2. `main` 是正式基线，不允许在基线未稳定时直接派生大量任务
3. 目录结构一旦确认，就是正式结构，agent 不得擅自重构或改目录
4. 每个任务必须明确允许修改范围与禁止修改范围
5. 所有重要决策、review、escalation 必须记录到 `docs/`
6. 先看索引文件，再按当前阶段按需读取文档
7. 修改后必须执行最相关验证
8. Gemini 在 review 阶段只负责审查，不直接修改代码

## 任务流转
标准流转顺序：
1. 基线准备
2. 需求澄清
3. 任务编排
4. 工作区派发
5. 执行开发
6. Review / Escalation
7. 人工确认与合并

默认责任人：
- 基线准备 → 用户 / 主控 agent
- 需求澄清 → Claude（默认）
- 任务编排 → Codex（默认）
- 工作区派发 → 主控 agent
- 执行开发 → Codex / 指定 agent
- Review / Escalation → Gemini / Claude
- 人工确认与合并 → 用户 / 主控 agent

在被明确指定时，Claude、Codex、Gemini 都可以协助其他合理任务，但必须遵守当前边界与流程。

## 索引文件与按需读取
所有 agent 开始工作前：
1. 先读 `README.md`
2. 再读 `docs/INDEX.md`
3. 只读当前阶段和当前任务最相关的文件
4. 不要默认一次性读完整个仓库

## 并行开发规则
### 默认并行度
- 小任务：1–2 个 agent 即可
- 中任务：2–3 个 agent
- 大任务：3–5 个 agent
- 总 agent 数不要超过 5 个

### 任务与 agent 规则
- 小任务不要过度拆分 agent
- 一个 agent 可以执行多个任务，但这些任务应属于同一模块、同一目标，或文件范围高度相关
- 每个 agent 都必须有清晰的任务简称、负责范围、目标说明
- 每个任务都必须明确允许修改范围与禁止修改范围

### 分支 / worktree 规则
- 只有在 `main` 已提交、基线稳定后，才创建 branch / worktree
- 每个工作区都应从 `main` 复制完整基线
- 有依赖关系的任务必须串行，不得盲目并行
- 多个 agent 不应同时修改同一核心共享文件

## 默认执行策略
- 如果用户已经明确指定任务，默认当前工作区就是有效工作区
- 不要每次都先检查 branch / worktree / git status
- 只有在以下情况，才进入严格预检：
  - 用户要求并行开发
  - 需要新开 branch / worktree
  - 涉及共享核心文件
  - 准备提交 / 合并 / review
  - 发现冲突风险或目录状态异常

## 升级规则（Escalation）
满足以下任一条件，必须升级：
- 同一问题连续失败 2 次
- 连续 2 次得到相似错误结果
- 需要跨模块重新设计
- 影响范围不清楚
- 当前任务超出授权范围

升级路径：
- 架构 / 复杂 bug / 重规划 → Claude
- 大上下文 / 影响面分析 → Gemini
- 分析完成后，修复动作默认回到 Codex 或指定执行 agent

## 交接要求（Handoff）
升级时必须记录：
- 当前任务简称
- 当前 agent
- 问题描述
- 已尝试方案
- 当前错误信息
- 涉及文件
- 升级给谁
- 希望接手方完成什么

## 禁止事项
- 不要在 `main` 还是空或未整理好的情况下直接开大量分支
- 不要让 agent 无边界修改整个项目
- 不要让有依赖关系的任务盲目并行
- 不要跳过 review 直接合并
- 不要在当前工作区有未处理改动时随意合并别的分支

## 验证规则
按任务相关性执行：
- lint
- typecheck
- test
- build
- 必要时手动验证
