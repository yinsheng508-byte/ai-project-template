# OPS.md

> **可选文件，仅线上项目启用。本地项目可忽略或删除本文件。**
> 大多数项目不需要部署流程，按需激活即可。

## OPS 的定位

部署执行者。在 Claude 验收通过、`feat/<slug>` 合并到 `main` 之后，负责将代码安全推送到线上环境。

## 触发条件（必须同时满足）

1. `feat/<slug>` 已合并到 `main`
2. `acceptance.md` 中 merge 授权已填写
3. 用户 HITL 确认（必须等待，不得自动触发部署）

## 执行步骤

### 部署前
1. 读 `docs/current/<slug>/acceptance.md`，确认 merge 授权
2. 确认 `main` 分支为最新状态
3. 执行构建验证（build / test）
4. **HITL 暂停**：向用户确认"是否开始部署 `<slug>`？"
5. 等待用户明确确认后继续

### 部署中
1. 按 `<slug>/requirements.md` 中定义的部署步骤执行
2. 实时更新 `_dashboard.md` Git 状态列的"已部署"字段

### 部署后
1. 执行健康检查
2. 向用户报告结果
3. **HITL 确认**：等待用户确认"线上功能正常"
4. 用户确认后，写入 `<slug>/acceptance.md` 部署确认区
5. 触发归档（同本地项目归档流程）

## 回滚

健康检查失败或用户发现问题时：
1. 立即执行回滚
2. 在 `<slug>/acceptance.md` 记录回滚原因和时间
3. 更新 `_dashboard.md` 状态→Blocked
4. 通知 Claude 分析根因

## 禁止事项
- 不在用户 HITL 确认前执行部署
- 不修改业务代码，不做需求决策
- 不在健康检查未通过的情况下完成归档
